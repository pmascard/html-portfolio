The Detailed History (wip)

I am a product of the growth of Information Technology.

My father was an electronics wizard and I grew up surounded by his projects. Most notably:

  • A self build kit version of a Pong TV console
  • Some device built with a ribbon cable conneted remote interface similar to a Sinclair Cambridge calculator
  • A hand built metal detector
  • Voltage regulators

Eventually, this led to a Commodore PET computer and eventually to an Apple II. Both of which were connected to instruments he'd designed to improve the manufacturing process for his business.

Whilst his passion did not directly drive my own learning (if only I'd started to learn to code on the Commodore PET when I was 9 or 10!) it did intorduce me to computing and I remember keying in pages and pages of magazine provided basic code into the PET so that I could play that particular game, if I didn't make a mistake.

I studied electronics and became a systems engineer, but always found the writing of code more interesting. I my early days on Naval defense systems, I would need to write code that analysed test output and printed reports to verify performance. These programs were written on paper templates that the typing pool would key in to the computers and were output to paper tape. A fond memory is of debugging the resulting code by finding the failure section and splicing in hand punched characters to replace the errors.

This led to a move to the commercial world where I would learn to program early PCs in solving small business problems that mainframe development was deemed too expensive to use. Happy days of fixing PC's, printers and building networks as well as writting local 4GL database applications.

In the early days there was no sharing of data, other than moving floppy disks around. During this period I learnt analysis of requirements, code structuring, procedural flow, user interface design, the value of building libraries of shared utilities (code reuse). Unfortunately, I never progressed to deeper languages, prefering to design forms using a tool over using code. The process was termed Rapid Application Development and this iterative code, test, demo, fail, learn, fix, retest, approve, deploy cycle was in use long before I heard the term Agile.

I have been solving business functionality problems using logic and technology throughout my entire career. I was exposed to personel computing before the age of 11 and have enjoyed being an interface between the hardware and the business community by developing solutions to solve business problems.

These solutions have been in terms of applications, data analysis, reporting, data management and governance, data transformation and warehousing. All of this has been with a passion for doing the job correctly.

Of late the expanse of technology tooling and the exponential growth and diversification of businesses providing those, all very similiar but marketed by huge budgets to be "the only tool your business will ever need" has confused the delivery of IT solutions.

I wanted to return to "simpler" times and decided that I needed to specialise again. My chosed focal points for specialisation were all about being able to recreate the precision and seriousness I took with my early development footsteps. Since the late 90s technology has improved massively, but the practices used to code, and deliver excellence, has not. I will apply what makes me tick to:

  • Web development practices
  • Modern architecture for Continuous Integration
  • Cross platform applications
  • Data management disciplines and practices
    • Microsoft SQL Server
    • PostgreSQL
    • Microsoft Power BI
    • Data quality importance, assessment and monitoring
  • Release management, version control, continuous integration
  • Documentation
